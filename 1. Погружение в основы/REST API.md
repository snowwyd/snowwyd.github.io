## Что такое API?
---
**API (Application Programming Interface)** — это, если грубо, **"официант"** между двумя программами.

*   **Ситуация:** Ты (клиент) в ресторане. Ты хочешь заказать еду (получить данные или выполнить действие).
*   **Проблема:** Ты не можешь пойти на кухню (в базу данных или другую программу) и самому все приготовить. Это хаос!
*   **Решение:** Ты зовешь официанта (API). Ты говоришь ему: "Принеси, пожалуйста, пиццу Маргариту" (это твой *запрос*). Официант знает, как передать этот заказ на кухню, дождаться пиццы и принести ее тебе (это *ответ*).

**Проще говоря:** API — это набор четких правил, как одной программе "общаться" с другой, чтобы что-то получить или сделать. Когда ты используешь приложение на телефоне, оно через API связывается с сервером, чтобы показать тебе ленту новостей, отправить сообщение и т.д.

## Что такое REST API?
---
Теперь давай представим, что наш "официант" (API) работает по очень строгому и логичному регламенту. Этот регламент и есть **REST (Representational State Transfer)**.

REST — это не инструмент и не язык, а **архитектурный стиль**, набор правил и соглашений о том, как создавать API, чтобы они были простыми, понятными и надежными.

**Основные правила REST API:**

1.  **Всё — ресурсы.** Любая информация — это ресурс. Например: пользователь, товар, заказ, статья. Каждый ресурс имеет свой уникальный адрес (URL). Например:
    *   `/users` — коллекция всех пользователей.
    *   `/users/123` — конкретный пользователь с ID=123.
    *   `/products` — коллекция всех товаров.

2.  **Использование HTTP-методов.** Мы "управляем" ресурсами с помощью стандартных команд HTTP, которые ты уже видел в браузере:
    *   `GET` — **получить** ресурс. ("Принеси мне данные о пользователе 123"). Безопасный, только для чтения.
    *   `POST` — **создать** новый ресурс. ("Создай нового пользователя с такими-то данными").
    *   `PUT` / `PATCH` — **обновить** существующий ресурс. ("Измени данные пользователя 123").
    *   `DELETE` — **удалить** ресурс. ("Удали пользователя 123").

3.  **Stateless (Без состояния).** Сервер не помнит твои предыдущие запросы. Каждый запрос должен содержать всю информацию, необходимую для его выполнения. Это как общение с очень забывчивым, но честным клерком: ты каждый раз называешь свое имя и говоришь, что тебе нужно, с нуля.

4.  **Ответы в стандартных форматах.** Сервер отвечает тебе данными в форматах, которые легко читать и программам, и людям. Самый популярный — **JSON** (JavaScript Object Notation). Он очень похож на структуры в Go.
    Пример JSON для пользователя:
    ```json
    {
      "id": 123,
      "name": "Иван Иванов",
      "email": "ivan@example.com"
    }
    ```

**Итог по REST API:** Это API, которое использует:
*   **URL** для указания *чего* мы хотим (ресурс).
*   **HTTP-методы** для указания *что* мы хотим с ним сделать (действие).
*   **JSON** для обмена данными.

## Что такое фреймворк?
---
Если язык программирования (как Go) — это набор базовых инструментов (молоток, пила, гвозди), то **фреймворк** — это готовый "станок" или "конструктор", который ускоряет сборку сложных вещей.

*   **Без фреймворка:** Тебе пришлось бы самому писать код для разбора каждого HTTP-запроса, определения его метода, извлечения данных и т.д. Это долго и сложно.
*   **С фреймворком:** Фреймворк уже содержит готовые "кирпичики" для создания веб-сервисов. Ты просто говоришь: "На адрес `/users` на метод `GET` должен выполняться вот этот мой код". Фреймворк берет на себя всю рутину.

**Популярные фреймворки для Go:**
*   **Gin** - очень быстрый и простой, идеален для начинающих.
*   **Echo** - еще один популярный и минималистичный фреймворк.
*   **Chi** - легковесный фреймворк для построения маршрутов.

Мы начнем с **Gin**, потому что он очень дружелюбен к новичкам.
