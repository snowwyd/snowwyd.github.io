## Что такое gRPC?
---
Если REST API — это **общение через почту** (отправляешь письмо, ждешь ответ), то gRPC — это **телефонный разговор** (непрерывное, быстрое общение).

**gRPC (gRPC Remote Procedure Call)** — это современный фреймворк для общения между программами, где одна программа может вызывать функции другой программы, даже если они работают на разных компьютерах, как если бы они были локальными.

## Чем gRPC отличается от REST API?
---

| Аспект                   | REST API         | gRPC                        |
| ------------------------ | ---------------- | --------------------------- |
| **Формат данных**        | JSON/XML (текст) | Protocol Buffers (бинарный) |
| **Скорость**             | Медленнее        | Быстрее (меньше данных)     |
| **Сложность**            | Проще для начала | Сложнее в настройке         |
| **Поддержка в браузере** | Полная           | Ограниченная                |

## Ключевые компоненты gRPC
---
**Protocol Buffers (protobuf)** — это язык для описания структур данных и сервисов. Представь, что это **чертеж** или **контракт**, который определяет:
- Какие данные можно передавать
- Какие функции можно вызывать
- Какие параметры они принимают

**HTTP/2** — современная версия HTTP, которая позволяет:
- Мультиплексирование (много запросов в одном соединении)
- Бинарную передачу (вместо текста)
- Приоритизацию запросов

## Как работает gRPC?
---
Представь, что ты хочешь заказать пиццу:

**REST подход:**
1. Ты пишешь письмо: "Хочу пиццу с пепперони"
2. Отправляешь на адрес: `POST /orders`
3. Ждешь ответ: "Ваш заказ №123 принят"

**gRPC подход:**
1. Ты звонишь в пиццерию и говоришь: "ЗакажиПиццу(тип: 'Пепперони')"
2. Ты сразу получаете ответ: "НомерЗаказа: 123"

**Преимущество gRPC:** Типизация и скорость. Компилятор проверяет, что ты передаешь правильные параметры, и бинарный формат делает обмен быстрее.

## Когда использовать gRPC, а когда REST?
---
**Используй gRPC когда:**
- Высокая производительность критична (микросервисы, внутренняя коммуникация)
- Нужна строгая типизация между клиентом и сервером
- Работаешь со стриминговыми данными (видео, аудио, реальное время)
- Клиенты и серверы на разных языках, но нужен единый контракт

**Используй REST когда:**
- Нужна простота и быстрое прототипирование
- Клиенты - веб-браузеры или мобильные приложения
- Уже есть REST API и нужно поддерживать обратную совместимость
- Команда не знакома с gRPC
